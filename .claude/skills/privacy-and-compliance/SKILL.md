## What this covers
The non-negotiable rules that apply across every script, template, and
example in this repo, because the subject matter is a school and its
students.

## Purpose of the project
This toolkit automates marketing content prep for Silverleaf. Because the
raw material is often stories/photos of minors, privacy has to be enforced
structurally, not just remembered.

## The rules
- **Never include a real student's name, photo, or identifying story** in
  any template, example, generated brief, or committed sample data —
  without documented parental/guardian consent on file. This repo is
  public, so anything committed here is public.
- All sample/example data in this repo (`content_calendar/calendar.csv`,
  `examples/example_filled_video_script.md`) must be obviously synthetic
  (e.g. "Aisha M.", generic grade levels) — never real Silverleaf student
  data, even as a "just an example."
- Never invent engagement numbers, statistics, or facts anywhere in this
  project. If data is missing, flag it (`NEEDS INPUT: ...`) — don't guess.
- Never delete calendar rows or performance-log history.
- Always ask before publishing or sending any generated content outside
  this workspace — nothing here auto-publishes anywhere.

## Common tasks
- Adding a new example or sample calendar row: use a clearly fictional name
  and check it reads as obviously synthetic, not like a real student.
- Reviewing a generated brief before it goes to a filmer/designer: confirm
  every real student reference in the underlying notes has a signed consent
  form on file — this check happens outside this repo (consent forms
  aren't stored here) but must happen before publishing.
- If asked to pull in real content (e.g. from an existing script or notes
  file) to seed an example, genericize it first — replace real names/details
  with placeholders rather than committing the original.

## Gotchas
- "It's just an example/test" is not an exception — treat every file in
  this repo as if it will be public forever, because it is public.
- Consent status isn't tracked in `calendar.csv` — don't assume a `status`
  of `planned` or `posted` says anything about consent. That check is
  separate and manual.
